<objective>
Build the LangGraph RAG orchestration graph that implements the retrieve-grade-generate flow.

Purpose: This is the core intelligence layer of the system. It takes a user query, retrieves relevant document chunks from ChromaDB, grades them for relevance using an LLM call, and generates a grounded Chinese answer. When no relevant chunks are found, it produces a polite fallback response. This graph is invoked by the FastAPI endpoint (Plan 02).

Output: A compiled LangGraph StateGraph exposed as `rag_graph` from `src/rag/graph.py`, with unit tests validating graph compilation and node behavior.
</objective>

**`retrieve(state: RAGState) -> dict`:**
- Get vectorstore via `get_vectorstore()`
- Create retriever with `vectorstore.as_retriever(search_kwargs={"k": settings.RETRIEVAL_TOP_K})` (default k=4)
- Invoke retriever with `state["query"]`
- Return `{"documents": docs}`

**`grade_documents(state: RAGState) -> dict`:**
- Initialize `ChatQwen(model="qwen-plus", temperature=0)` for deterministic grading
- For each document in `state["documents"]`:
  - Format GRADER_HUMAN_PROMPT with query and document.page_content
  - Send [SystemMessage(GRADER_SYSTEM_PROMPT), HumanMessage(formatted)] to LLM
  - Parse response: if "yes" in response.content.lower(), keep the document
- Collect filtered docs into a list
- Return `{"filtered_documents": filtered_docs, "has_relevant_docs": len(filtered_docs) > 0}`

**`generate(state: RAGState) -> dict`:**
- If not `state["has_relevant_docs"]`, return `{"generation": FALLBACK_RESPONSE}`
- Concatenate filtered_documents' page_content with double newlines as context string
- Initialize `ChatQwen(model="qwen-plus", temperature=0.3)` for generation (slight creativity for natural Chinese)
- Format GENERATOR_SYSTEM_PROMPT with context
- Send [SystemMessage(formatted_system), HumanMessage(GENERATOR_HUMAN_PROMPT.format(query=state["query"]))]
- Return `{"generation": response.content}`

CRITICAL: Do NOT mutate state dict. Always return a new dict with only changed keys. This is enforced by LangGraph's StateGraph.

CRITICAL: Use `ChatQwen` from `langchain_qwq`, NOT `ChatTongyi` from `langchain-community`. ChatTongyi is deprecated for Qwen3+ models.

CRITICAL: For `ChatQwen`, pass `model="qwen-plus"` (not `model_name`). The `langchain-qwq` ChatQwen class uses `model` as the parameter name.

Create `src/rag/graph.py`:

```python
from langgraph.graph import StateGraph, START, END
from src.rag.state import RAGState
from src.rag.nodes import retrieve, grade_documents, generate


def build_rag_graph():
    """Build the RAG graph: retrieve -> grade_documents -> generate."""
    graph = StateGraph(RAGState)

    # Add nodes
    graph.add_node("retrieve", retrieve)
    graph.add_node("grade_documents", grade_documents)
    graph.add_node("generate", generate)

    # Define edges: linear flow retrieve -> grade -> generate
    graph.add_edge(START, "retrieve")
    graph.add_edge("retrieve", "grade_documents")
    graph.add_edge("grade_documents", "generate")
    graph.add_edge("generate", END)

    return graph.compile()


# Module-level compiled graph instance for import by FastAPI endpoint
rag_graph = build_rag_graph()
```

The graph uses a linear flow (retrieve -> grade -> generate) because the grade node already handles the "no relevant docs" case by setting `has_relevant_docs=False`, and the generate node checks this flag to return a fallback. This avoids unnecessary conditional edge complexity while still handling the fallback path correctly.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User query -> RAG graph | Untrusted user text enters the LangGraph pipeline as query |
| Retrieved docs -> LLM prompt | Document content from ChromaDB is injected into LLM prompts |
| LLM response -> output | LLM-generated text is returned to the user |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation |
|-----------|----------|-----------|----------|-------------|------------|
| T-02-01 | Injection | retrieve node | HIGH | mitigate | ChromaDB similarity search treats query as embedding vector, not SQL — no injection vector. However, query string is also passed to LLM prompts in grade/generate nodes. Mitigate by using structured prompt templates with `{query}` placeholder (not f-string concatenation), ensuring LLM sees query as data within a system-prompt-defined role. |
| T-02-02 | Injection | generate node | HIGH | mitigate | Retrieved document content is injected into GENERATOR_SYSTEM_PROMPT. A poisoned document could contain prompt injection text. Mitigate by: (1) system prompt explicitly instructs LLM to only answer based on document content, (2) grade_documents filters irrelevant chunks reducing attack surface, (3) document content is placed in a clearly delimited section of the prompt. |
| T-02-03 | Information Disclosure | generate node | MED | mitigate | LLM may hallucinate or leak content from documents unrelated to the query. Mitigate by: (1) relevance grading filters chunks before generation, (2) system prompt instructs "only answer based on provided documents", (3) FALLBACK_RESPONSE is returned when no relevant docs found instead of LLM free generation. |
| T-02-04 | Denial of Service | grade_documents node | MED | accept | Each query triggers N LLM calls for grading (one per retrieved chunk). With k=4, this means 4 grading calls + 1 generation call = 5 LLM calls per query. Acceptable for internal single-tenant MVP. Rate limiting is applied at the API layer (Plan 02). |
| T-02-05 | Spoofing | DASHSCOPE_API_KEY | HIGH | mitigate | API key loaded from environment variable, never hardcoded. Docker Compose uses env_file directive. Key is not logged or exposed in error messages. Node functions do not catch and re-raise exceptions with key details. |
</threat_model>
