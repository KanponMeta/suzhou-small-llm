```python
from pydantic import BaseModel, Field
from typing import Literal, Optional, Dict

class TestCase(BaseModel):
    """A single evaluation Q&A test case per 数据集提交指南.md spec."""
    id: str = Field(..., description="Unique identifier for the test case")
    task_type: Literal["chat:text"] = Field(default="chat:text", description="Task type — always chat:text for this project")
    category: str = Field(..., description="Category derived from document content/topic")
    user_prompt: str = Field(..., description="The question to ask the model")
    answer_type: Literal["free_form"] = Field(default="free_form", description="Answer type — always free_form for this project")
    options: Optional[Dict[str, str]] = Field(default=None, description="Not used for free_form type")
    correct_answer: str = Field(..., description="Ground-truth answer derived from document chunk")

    # Metadata (not serialized to evaluation_data.json, used internally for traceability)
    source_document: Optional[str] = Field(default=None, exclude=True, description="Source document filename for traceability")
    source_chunk_id: Optional[str] = Field(default=None, exclude=True, description="ChromaDB chunk ID for traceability")

class EvaluationData(BaseModel):
    """Top-level structure for evaluation_data.json per 数据集提交指南.md spec."""
    test_cases: list[TestCase]
```

Create `src/dataset/generator.py` implementing Q&A pair generation:

1. **LLM Setup**: Instantiate `ChatQwen` from `langchain_qwq` with `model="qwen-long"` (optimized for long-context document processing, cheapest per token per CLAUDE.md stack guidance). Use `DASHSCOPE_API_KEY` from environment/config.

2. **Core function `generate_qa_from_chunks(chunks, source_name, category)`**:
   - Input: list of document chunk texts (strings), source document name, category string
   - For each chunk (or batch of related chunks from same document):
     - Build a prompt that provides the chunk text as context and instructs the LLM to generate 2-3 Q&A pairs
     - The prompt MUST enforce grounding: "仅基于以下文档内容生成问答对，不要使用任何外部知识。如果文档内容不足以回答某个问题，不要生成该问答对。"
     - The prompt MUST instruct the LLM to output JSON array of objects with keys: `user_prompt`, `correct_answer`
     - Use `ChatQwen` with structured output (Pydantic model or JSON mode) to get clean JSON back
   - Parse LLM output into `TestCase` objects:
     - `id`: generate as `f"qa_{uuid.uuid4().hex[:8]}"` for uniqueness
     - `task_type`: "chat:text" (hardcoded)
     - `category`: derived from source document name or content topic (passed as parameter)
     - `answer_type`: "free_form" (hardcoded)
     - `options`: None
     - `source_document`: set to source filename (excluded from JSON export but kept for traceability)
     - `source_chunk_id`: set to chunk identifier if available
   - Return: `list[TestCase]`

3. **Orchestration function `generate_qa_pairs(vectorstore)`**:
   - Input: ChromaDB Chroma vectorstore instance (from `langchain_chroma.Chroma`)
   - Retrieve ALL document chunks from the collection using `vectorstore._collection.get()` (returns all stored documents with metadata)
   - Group chunks by `source` metadata field (each source document's chunks together)
   - For each source document group:
     - Derive `category` from the source filename (strip extension, use as category — e.g., "企业规章制度.pdf" becomes "企业规章制度")
     - Call `generate_qa_from_chunks()` with the chunk texts, source name, and category
   - Collect all TestCase objects into a flat list
   - Return: `EvaluationData(test_cases=all_test_cases)`

4. **Prompt template** (embed in generator.py as a constant):
```python
QA_GENERATION_PROMPT = """你是一个专业的评测数据集生成助手。请仅基于以下文档内容生成高质量的中文问答对。

严格要求：
1. 问题和答案必须完全基于提供的文档内容，禁止使用任何外部知识
2. 答案必须可以从文档中直接找到或推导出来
3. 问题应该多样化，覆盖文档中的不同知识点
4. 每个问答对的答案应该完整、准确

文档内容：
{chunk_text}

请生成 2-3 个问答对，以 JSON 数组格式输出：
[
  {{"user_prompt": "问题内容", "correct_answer": "基于文档的准确答案"}}
]

仅输出 JSON 数组，不要添加其他说明文字。"""
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Document chunk content -> LLM prompt | Untrusted document text is injected into the generation prompt — potential prompt injection vector |
| LLM output -> TestCase parsing | LLM returns JSON that is parsed into Pydantic models — malformed output must be handled |
| ChromaDB -> generator | Vector store data is read-only in this flow but could contain adversarial content from ingestion |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | QA_GENERATION_PROMPT | mitigate | Document chunk text is inserted into a fixed prompt template with clear delimiters. The prompt instructs "仅基于以下文档内容" to constrain output. Chunk text is treated as data, not instructions — no f-string interpolation of user-controlled template variables. Use `.format(chunk_text=text)` with a single known placeholder. |
| T-03-02 | Spoofing | LLM structured output | mitigate | All LLM output is parsed through Pydantic v2 `TestCase` model validation. Invalid fields, wrong types, or missing required fields are rejected by Pydantic. JSON parsing failures are caught with try/except and logged, chunk is skipped. |
| T-03-03 | Denial of Service | generate_qa_pairs | mitigate | Large document collections could trigger excessive LLM calls. Implement chunk count logging. The function processes chunks sequentially per document, with error handling to skip failed chunks rather than crash. Rate limiting is inherent in DashScope API quotas. |
| T-03-04 | Information Disclosure | correct_answer field | accept | Generated answers may expose sensitive document content. This is expected behavior — the dataset is generated from enterprise documents for internal use. The ZIP export is a deliberate data extraction feature. |
</threat_model>
